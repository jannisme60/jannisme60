## How I Think About Projects

I usually start from a **product or workflow problem**, not from a technology.

For each serious project, I try to document:

- the problem or opportunity
- the target user or use case
- the MVP scope
- the core product decisions
- the technical approach
- the limitations and trade-offs
- possible next iterations

The goal is not just to show that something works, but to make clear **why it was built that way**.
